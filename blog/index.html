6 Key QWERTY or How I learned to love bitwise operators in Javascript.

 9th April, 2017

This will be the first part of a two-part blog post on my 6-Key-QWERTY project. The idea behind this project is to build a six key chording keyboard that is portable and wireless. A keyboard that you could touch-type with while standing; while sitting; while laying down.

This is not a new idea for me and I don't have any illusions that this will be a popular project or method of typing outside of myself but I want to complete this project while at the Recurse Center because it is a project I've visited 2-3 times in the span of ~10 years and only right now do I have the technical skills to implement a physical prototype fully.

Getting to the JavaScript portion of this project and why this is a two-part blog post, the project has a front-end component to teach people how to type using my mapping, built in JavaScript. The project also has a physical component built using C, an Arduino micro-controller, a Bluetooth radio and some push buttons.

At this point I have both implemented the basic keyboard mapping and some visual UI/UX cues on top of an svg image of a standard QWERTY keyboard to assist people while they learn the mapping. As this is a chording keyboard design, for most characters/numbers/symbols, you will have to press more than one key for one character to be written to the screen.

Around 4 years ago I struggled with how to implement this in a very naive way (inside MS excel btw as I was the most experienced with Excl VBA at that time because I used it for work on a daily basis). I didn't care about the order of the multiple keys being pressed, just that you need to, for example, hold 's' & 'f' together before releasing either so that the letter 'a' would be written to the screen, I captured key down and key up events, and inside a dictionary created both 'sf' and & 'fs' keys that had 'a' as a value. My chording keyboard was born. I found it hard to learn, moved on to my next job and end of story.

Except that it wasn't. I had totally forgotten about this project up until a very serendipitous conversation with some of my batch-mates regarding ergonomic and chording keyboards. The six key keyboard was back in my mind.

For the learning user interface, from which you can test out the mapping with a normal keyboard while using the home-row (s,d,f,j,k & l), I decided to use JavaScript, an SVG image of a keyboard and D3.js to add some interactivity to it.

For the state representation of what buttons have been pressed (and the user interface representing this), I decided to use individual bits and JavaScript bitwise operators. I'd had heard of config flags, registers, etc. in a lot of low level code but had never personally used individual bits and operations on them in any language. My decision was also inspired by having to re-implement the mapping part (no UI) of the project in C to be able to use an Arduino micro-controller.

With a couple of bitwise operator bugs under my belt, I can now appreciate the simplicity and relevancy of bits being used for this specific use-case. Below I will give a brief example of how I'm using bitwise operators to control the UI to assist the user in learning the keyboard mapping.

I will give a concrete example of the mapping: to type 't' you need to press 'f', 'j' & 'l'. To assist the user in seeing what characters they could type after holding down the first key I wanted to 'blur out' the 'impossible' key combinations by reducing the opacity of those keys. The code below assists in doing this.

      var exclude = d3.select(svg).selectAll(".control,"+selectorString).nodes().filter( function(el) {
        return ((parseInt(el.id.split("-")[2]) & bit) == 0);
      });
      d3.selectAll(exclude).style("opacity",0.25);

Above, I'm filtering out the keys that return a 0 when I bitwise 'AND' the binary representation of the key (more specifically the combination of keys required to type it) with the bitwise representation of the currently pressed key. In less mathematical terms, I am only showing the keys that have the currently pressed key in their key combination. Using bit arithmetic is the most elegant, concise and appropriate way to represent this problem; as much as it was not the most intuitive way at first.

When I was satisfied that the code above was working correctly, I wanted to go a step further: I wanted to change the fill color of the remaining keys with the color of the remaining key/keys that would need to be pressed to type that letter. This meant that the background needed either 1 or 2 colors as there are 3-key combinations in the mapping. The below code accomplishes this.

var shiftBits = parseInt(this.parentNode.id.split("-")[2]) - bit;
          shiftBits = shiftBits == 0 ? bit : shiftBits;
          var shiftBits2 = shiftBits;
          keyCombinations[shiftBits2] = [];
          var place = 0;
          while (shiftBits > 0) {
            if ( (shiftBits & 1) == 1 ) {
              var tmp = d3.select(svg).selectAll(".home").nodes().filter( function(el) {
                return (parseInt(el.id.split("-")[2]) == Math.pow(2,place))
              });
              keyCombinations[shiftBits2].push(d3.selectAll(tmp).select("path").style("fill"));
            }
            shiftBits >>>= 1;
            place ++;
          }

What the code above does is look for the remaining colors (1 or 2 for 2 or 3 key combinations respectively) needed to press a specific key. I leave you to analyze these two code snippets and think alternative ways to implement this. There are other methods and data structures one could use but trying to stay as low overhead as possible in preparation for the Arduino component of the project yielded a new found understanding, appreciation and enjoyment of using bits and bitwise operators. I would have never thought I would use this, let alone to control UI elements with JavaScript.

Stay tuned for Part II of this post where I go through the process of implementing this design physically.

6 Weeks at the Recurse Center

 2nd April, 2017

This is a blog post about reaching my halfway point at the Recurse Center

How to start? I've put it off for too long. So how to start? By starting. Hopefully after this I will be more consistent and hence this should get easier.

My coding output in the past 6-7 weeks, the first half of my 12 week batch has been great. Have made it a habit to try to make progress on something every day. What have I learned in the past 6 weeks? What can I improve for the second half? Many questions. Let's try and answer some of them.

In the months prior to RC I spent a lot of time reading: catching up on new tech trends, learning about new technologies and frameworks and trying to better understand the tech landscape. This was an invaluable time as I was introduced to the DevOps discipline and that opened my mind to instances in my past where professionally employers/teams were doing things right, and times when they were doing things wrong. What I did less of in those couple of months was code. That changed since I started at RC.

Reading time was over, it was coding time. The structure at RC really encouraged me to just code. Code on current projects, seek out new ones, wrap up older projects. I tuned out other things to assist me in coding as much as possible. Read less tech news, less Twitter; coded more, much more.

As much as my output and general plan for RC has worked out well, I have also been able to embrace it's serendipity while still applying that to Coding More . One week into my batch I started contributing to somebody else's project and squashing bugs, submitting pull requests and adding functionality to a web app. In another instance I got outside my wheelhouse and interests and worked on a game for ~5 days with a partner for a game jam. That gave me a chance to write more Lua code (which I'm always up for), but the most rewarding part was working on a common goal within a limited time-frame with a hard deadline. The game turned out great and a lot of fun to play. Left to my own devices, I would have not worked on a game.

On every project I've worked on, and any advice/help I've provided to people, I've framed the situation from a DevOps perspective. I see that a lot of people here are not interested in the 'plumbing'/infrastructure side of things, while I am. Many person/hrs are spent trying to resolve issues of dependency hell and other operations' tasks. I do believe this prevents or slows down collaborating on projects at RC. For all my projects I've taken the responsibility to create repeatable development (and production if applicable) environments to reduce operational toil. I've also provided help and advice to others regarding DevOps principles. By working on this and containerizing applications I've improved my practical experience with Docker and related technologies considerably.

So any changes for the next 6 weeks? Yes and no. I don't plan on making any considerable changes on how I'm planning what projects to work on. I do have a backlog of projects so as I complete projects, the projects will change but my methodology will remain the same: work on 2-3 projects max at the same time; plan for the weeks ahead but be open and embrace serendipity and this will guide what projects I work on next.

Two things I do want to change a bit are presenting more and writing more. I want to try and post some type of blog-post, or quick write up once a week. The other thing is to present on something, anything, at least once a week (we have 5 minute presentations every Thursday). Although it might seem that this would mean coding less, it might have the opposite effect as having a project at a presentable stage once a week encourages me to code more so that it is ready for a presentation. I want to also present on DevOps topics. I want to introduce people to Docker, orchestration technologies, and why these are important in reducing toil and unnecessary operations' headaches.

As much as other people at RC are more experienced in certain topics/technologies/languages than I am, on the flip side I've realized that I have my own domain knowledge which I've gladly shared with others and plan to continue doing this. From my professional background, I've gotten pretty knowledgeable about databases which is a topic others want to learn more about. From my recent online courses and readings, I've also become knowledgeable about DevOps technologies and methodologies which I'm more than happy to rant/teach about.

Sometimes personal growth/learning is not linear. My experience at RC up until now has taught me this. I've slowly learned about many technical topics in the past years, through work or from personal learning, but now with that knowledge base, by Coding More I've been able to put many disparate skills/knowledge into practice and I feel like I've learned so many things in the past 6 weeks. I truly feel I'll learn even more in the coming 6 weeks.
